AI-BASED ROAD HAZARD DETECTION
Complete Backend Architecture & Database Schema
24-Hour Hackathon Reference Document
1. System Overview
The backend is a FastAPI Python application that sits between two React frontends: the Testing Dashboard (where users upload video/image files) and the Admin Panel (command centre for authorities). Since there is no live CCTV hardware, every piece of data originates from a user-uploaded file. The backend processes frames from that file through a 6-layer AI pipeline, persists results in a relational database, and streams annotated frames and alerts in real-time over WebSocket.
1.1 Six-Layer Processing Pipeline
1 — API Layer: 
FastAPI WebSocket (/ws) + REST (/api/*). Entry point for all traffic from both frontends.
2 — Input Processing: 
Decodes base64 frame, resizes to 640×480, applies night-mode enhancement if brightness < threshold.
3 — AI Model Layer: 
Three parallel YOLOv8 threads: Pothole module, Animal module, Accident module. Each returns bounding boxes and confidence scores.
4 — Feature Extraction: 
Per-detection enrichment: severity classification, behavior tagging, speed estimation, road-zone check, geo-tag attachment.
5 — Multi-Hazard Fusion: 
Combines risks across all three modules into a single 0–100 risk score. Flags CRITICAL when multiple hazards co-occur.
6 — Output Layer: 
Three parallel outputs: annotated frame → Testing Dashboard, alert events → Admin Panel via WebSocket, all detections → database.
2. Database Design Principles
The database uses MongoDB (or PostgreSQL with JSONB) and is structured around a single central spine: upload_sessions → detections. Every other table is either a child of detections (1-to-1 module-detail tables) or a grouping layer built on top of detections (clusters, tracks, hotspots). This design means all cross-module analytics queries hit only the detections table, while module-specific queries join to their detail table.
Alert trigger rule: An alert row is created only when risk_score >= 70 OR the fusion layer sets combined_risk = CRITICAL. Low-severity detections are stored in detections but never generate an alert row.
3. Core / Shared Tables
These two tables are the root of the entire schema. Every other table in the system eventually joins back to one of them.
  CORE — upload_sessions  
upload_sessions
Purpose: One row per file uploaded to the Testing Dashboard. Replaces the concept of a live camera. All detections, alerts, analytics, tracks, and clusters are scoped to a session. If the same video is uploaded twice it produces two independent sessions.
Example use: User uploads 'highway_night.mp4'. A session row is created with status=pending. As frames are processed, processing_start is filled. When done, status=completed, processing_end is filled, total_frames=1240.

  CORE — detections  
detections
Purpose: Central fact table. Every AI detection from every module writes one row here first, then writes to its module-specific detail table. This allows all cross-module analytics (total count, severity distribution, risk over time) to query a single table without joining three separate detection tables.
Example use: YOLO detects a cow on frame 340 of a video with 85% confidence. One row goes into detections (type=animal, severity=high, risk_score=78). A second row goes into animal_details linked by detection_id. Because risk_score >= 70, an alert row is also created.

4. Pothole Module Tables
Four tables cover the three pothole USPs: severity classification (pothole_details), lifecycle tracking across frames (pothole_clusters + pothole_cluster_sightings), and automated reporting (pothole_reports).
  POTHOLE — Feature: Severity Classification  
pothole_details
Purpose: 1-to-1 extension of detections for pothole-specific attributes. Stores the raw measurements used to compute severity and the reason for the classification.
Example use: A pothole bbox is 180×60 px → area_px = 10800 (>8000 → high). It sits at road_position = center → severity bumped and reason stored as 'large area, road center position'. This row is created immediately after the detections row.
Severity logic: area_px < 2000 → low | 2000–8000 → medium | >8000 → high. Center lane position bumps severity up one level.

  POTHOLE — Feature: Lifecycle Tracking  
pothole_clusters
Purpose: De-duplicates the same physical pothole seen across many video frames. Without clustering, a 10-second clip of one pothole would create hundreds of separate detection rows. The cluster represents the single real-world pothole, while sightings record each frame it appeared in.
Example use: A pothole appears in frames 80–220 of a video (140 frames). One cluster row is created with first_frame=80, last_frame=220, sighting_count=140. Its severity starts as medium but the cluster tracks when it 'worsens' (bbox grows) and sets status=worsening.

5. Animal Module Tables
Five tables cover the animal USPs: behavior + region risk (animal_details), trajectory prediction (animal_track_sessions + animal_track_frames), and hotspot intelligence (animal_hotspots).
  ANIMAL — Feature: Behavior Classification + Region Risk  
animal_details
Purpose: 1-to-1 extension of detections for animal-specific attributes. Captures species, behavior state, whether the animal is inside the road zone, and whether night-mode preprocessing was applied.
Example use: A cow is detected crossing the road at night (behavior=crossing, in_road_zone=true, light_level=night, night_mode_applied=true). risk_score in detections is set to 85 → triggers an alert. The zone_label='road_center' tells authorities exactly how dangerous the position is.
Road zone definition: The middle 60% of frame height (y > 0.2×H and y < 0.8×H) is treated as the road zone for the hackathon. This threshold can be configured per session.

Example use: After processing, the backend computes that cells [3,4] and [3,5] in the 8×8 grid have the highest counts (20+ detections). These become the red hotspot areas on the heatmap. species='all' covers every animal; separate rows can be created per species.

6. Accident Module Tables
Five tables cover the accident USPs: severity scoring + hit-and-run + traffic disruption (accident_details), vehicle tracking for hit-and-run logic (vehicle_track_sessions + vehicle_track_frames), and predictive detection (accident_precursors).
  ACCIDENT — Feature: Severity Scoring + Hit-and-Run + Traffic Disruption  
accident_details
Purpose: 1-to-1 extension of detections for accident-specific attributes. Stores the multi-factor severity score, hit-and-run flag, and traffic density measurements before/after the event.
Example use: A collision between a truck and a bike is detected. vehicle_count=2, vehicle_types={truck:1, bike:1}, human_detected=true → severity_score=91 (critical). Within the next 30 frames the truck accelerates away → hit_and_run_flag=true, hit_and_run_confidence=0.88. Traffic density drops from 4.2 to 0.8 vehicles/frame → disruption_score=81%.

  ACCIDENT — Feature: Predictive Accident Detection  
accident_precursors
Purpose: Stores pre-collision warning events — anomalies in vehicle behavior (sudden stop, wrong direction, erratic path) that the system detects before an accident occurs. The led_to_accident field is filled retroactively by checking whether an accident detection followed within 60 frames.
Example use: A car makes a sudden stop (velocity drops from 12 px/frame to 0.3 in 3 frames) → precursor_type=sudden_stop, confidence=0.82. 18 frames later a collision is detected → led_to_accident=TRUE, related_detection_id is filled. This is the 'predictive' USP.

7. Analytics / Cache Tables
These tables avoid expensive re-aggregation on every Admin Panel request. They are written to incrementally as frames are processed and read by the Admin Panel every 30 seconds.
  ANALYTICS — Metric Cache  
analytics_cache
Purpose: Stores pre-computed aggregate metrics so the Admin Panel's metric cards and charts can be served instantly. Each row is one named metric for one session (or NULL session for global aggregates).
Example use: After every 10 frames, the backend upserts analytics_cache rows: metric_name='pothole_high_count', value=7, breakdown=null, session_id=X. The Admin Panel's 'Potholes Detected' card reads this row rather than COUNT(*) from detections.

  ANALYTICS — Per-Frame Stats  
frame_analytics
Purpose: One row per processed frame. Stores detection counts and max risk score for that frame. This is the source data for the Admin Panel's line chart ('risk score over time through the video') without needing to re-query detections.
Example use: The line chart query is simply: SELECT frame_number, max_risk_score FROM frame_analytics WHERE session_id = X ORDER BY frame_number. Rendering 1200 points from this lightweight table is instant.
